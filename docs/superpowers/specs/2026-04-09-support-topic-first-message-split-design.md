# Дизайн разделения первого сообщения в теме поддержки

## Цель

Изменить формат первого сообщения пользователя в теме поддержки так, чтобы при создании новой темы админ видел два отдельных сообщения:

1. служебную шапку с пользователем и Telegram ID;
2. отдельным следующим сообщением сам текст первого обращения.

После этого все дальнейшие сообщения пользователя в той же теме должны приходить обычным текстом без служебной шапки.

## Текущее состояние

Сейчас `handlers/support_bot.py` всегда отправляет в topic одно объединенное сообщение вида:

```text
Пользователь: @username
Telegram ID: 123

текст обращения
```

Это происходит и для первого сообщения в новой теме, и для последующих сообщений в уже существующей теме.

## Желаемое поведение

### Новая тема

Если для пользователя создается новая тема, бот отправляет в нее два сообщения подряд:

1. шапка:

```text
Пользователь: @ArchiveKiva
Telegram ID: 8425254259
```

2. текст первого обращения отдельным сообщением, например:

```text
12321321213
```

### Существующая тема

Если тема уже существует и рабочая, каждое следующее сообщение пользователя отправляется в нее только как чистый текст, например:

```text
вцйвйцвцйв
```

Без повторной шапки, без дополнительных данных и без кнопок.

### Пересозданная тема

Если старая тема была удалена админом, а бот при новой отправке восстанавливает ее заново, поведение новой темы повторяется:

1. сначала отдельная шапка;
2. затем отдельное сообщение с текстом нового обращения.

## Подходы

### Подход 1 — рекомендованный

При создании темы отправлять два сообщения, а при обычной переписке — одно текстовое.

Плюсы:

- визуально соответствует ожидаемому виду в чате админа;
- не требует отдельной схемы хранения флага `first_message_sent`;
- легко повторить при восстановлении удаленной темы.

Минусы:

- первое обращение создает два сообщения вместо одного.

### Подход 2 — одно сообщение с шапкой только в первый раз

Оставить первое обращение объединенным сообщением, а последующие отправлять только текстом.

Плюсы:

- меньше сообщений.

Минусы:

- не соответствует желаемому виду.

### Подход 3 — хранить отдельное состояние форматирования

Добавить в БД флаг, было ли уже отправлено первое служебное сообщение в текущей теме.

Плюсы:

- полностью явная модель состояния.

Минусы:

- лишняя сложность без необходимости;
- текущее наличие или пересоздание темы уже дает достаточно информации для выбора формата.

## Рекомендуемое решение

Использовать подход 1.

### Архитектура

- Текущая маршрутизация `пользователь -> support-бот -> forum topic -> админ` сохраняется.
- Решение строится на различии между новой темой и уже существующей темой.
- Создание новой темы и восстановление удаленной темы используют одинаковый формат: сначала шапка, затем текст сообщения.

### Изменения в `handlers/support_bot.py`

- Вынести отправку шапки в отдельный вызов `bot.send_message(...)`.
- Для новой темы:
  - создать или восстановить тему;
  - отправить шапку отдельным сообщением;
  - отправить текст обращения отдельным сообщением.
- Для существующей рабочей темы:
  - отправить только текст обращения.
- При `TelegramBadRequest` на старой теме:
  - создать новую тему;
  - снова отправить шапку;
  - затем отправить текст обращения.

### Данные

Новых таблиц и полей не требуется.

Факт новой темы определяется текущей логикой:

- тема отсутствует в `support_topics`, или
- тема числится в БД, но Telegram сообщает, что thread больше не существует.

## Поток данных

1. Пользователь отправляет сообщение support-боту.
2. Бот ищет тему пользователя в `support_topics`.
3. Если темы нет — создает новую тему, отправляет шапку, затем текст первого обращения.
4. Если тема есть и рабочая — отправляет только текст.
5. Если тема есть, но удалена — создает новую тему, обновляет mapping и снова отправляет шапку и текст.

## Ошибки и ограничения

- Если форум поддержки не настроен, ошибка остается текущей.
- Если создание новой темы упадет, сообщение не будет доставлено админу — это текущее ожидаемое поведение ошибки.
- Если пользователь без username, в шапке остается текущий fallback `без username`.

## Тестирование

Нужен только минимальный набор проверок:

- новая тема отправляет два сообщения: шапку и текст;
- существующая тема отправляет только текст без шапки;
- восстановленная после удаления тема снова отправляет шапку и текст;
- подтверждение пользователю не меняется.

## Критерии готовности

- В новой теме админ видит отдельную шапку и отдельное первое сообщение пользователя.
- В существующей теме админ видит только текст новых сообщений пользователя.
- После удаления темы новая созданная тема снова начинается с отдельной шапки.
- Кнопки и лишняя служебная разметка не появляются.
