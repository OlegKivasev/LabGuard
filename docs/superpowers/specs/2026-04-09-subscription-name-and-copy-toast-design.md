# Дизайн фикса имени подписки и toast-уведомления копирования

## Цель

Исправить два поведения в Mini App:

1. пользователь должен видеть в названии конфигурации только `🇫🇮 Финляндия`, без строк вида `🚀 Marz (labguard_olegkivasev) [VLESS - tcp]`;
2. после нажатия кнопки копирования должен показываться всплывающий toast, не сдвигающий интерфейс.

## Текущее состояние

- Новые ссылки уже частично нормализуются в `handlers/get_vpn.py` и `webapp.py`, но старое значение `subscription_url`, сохраненное в БД, может отдаваться пользователю как есть.
- Из-за этого Mini App показывает старый fragment из Marzban, даже если логика нормализации для новых ссылок уже добавлена.
- В `webapp.py` обработчик `copyBtn` использует блок `statusError`, поэтому сообщение о копировании вставляется в поток layout и двигает интерфейс.

## Желаемое поведение

- При любом показе подписки пользователю отображаемое имя должно быть только `🇫🇮 Финляндия`.
- Если в БД уже лежит старая ссылка с fragment от Marzban, она должна быть нормализована перед отдачей пользователю и сохранена обратно в БД в исправленном виде.
- После успешного копирования показывается зеленый toast `Ссылка скопирована` на 2-3 секунды.
- При неудаче копирования показывается красный toast с текстом ошибки или fallback-текстом.
- Основной layout экрана не должен смещаться.

## Подходы

### Подход 1 — рекомендованный

- Нормализовать `subscription_url` не только в момент создания, но и в момент чтения из БД.
- Перезаписывать исправленную строку обратно в БД.
- Для UI добавить отдельный toast-слой поверх страницы.

Плюсы:

- чинит и новые, и старые данные;
- не требует миграции БД;
- UI копирования становится стабильным и не прыгает.

Минусы:

- нужно затронуть и backend, и frontend.

### Подход 2 — исправлять только новые ссылки

- Оставить старые `subscription_url` как есть, нормализовать только вновь полученные ссылки.

Плюсы:

- минимальная правка.

Минусы:

- старые пользователи продолжат видеть `🚀 Marz (...)`.

### Подход 3 — разовая миграция БД

- Отдельно пройтись по базе и заменить все старые fragment заранее.

Плюсы:

- данные очищаются массово.

Минусы:

- лишняя сложность для небольшой задачи;
- все равно нужна защита на чтении, чтобы подобное не повторялось.

## Рекомендуемое решение

Использовать подход 1.

### Изменения в backend

- В `webapp.py` при чтении `stored_subscription_url` всегда прогонять строку через `_apply_subscription_display_names(...)`.
- Если после нормализации строка изменилась, сохранять ее обратно через `db.set_subscription_url(...)`.
- Для `vless://` fragment всегда должен становиться URL-encoded вариантом `🇫🇮 Финляндия`.
- Для `http(s)://` fragment всегда должен становиться `#🇫🇮 Финляндия`.

### Изменения в frontend

- Добавить отдельный toast-элемент поверх интерфейса.
- Завести небольшую функцию `showToast(message, type)`.
- Кнопка копирования вместо `statusError` должна использовать toast.

## Тестирование

- тест на нормализацию старого сохраненного `subscription_url` из БД;
- тест на показ пользователю только нормализованного имени;
- синтаксическая проверка `webapp.py`.

## Критерии готовности

- в UI больше не виден `🚀 Marz (labguard_olegkivasev) [VLESS - tcp]`;
- пользователь видит только `🇫🇮 Финляндия`;
- копирование не двигает layout;
- вместо встроенного красного блока появляется toast-уведомление.
